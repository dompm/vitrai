# Library picker rebuild -- before/after report

Generated by `scripts/build_swatch_library.py` (iteration 036), full rerun of the
5-manufacturer catalog (~1,381 candidate SKUs) with the vendored scored swatch-picker
(`scripts/swatch_picker.py`, research/delighting report 035) as the image-selection
step. Existing registry entries compared: **1333**. Final registry entries: **1265**.

The registry (`frontend/public/assets/glass_swatch_registry.json`) and the images
(`frontend/public/assets/catalog_images/`) are gitignored runtime data written into
the main checkout by the rebuild -- same precedent as the delighting-024 `-v2` drop.
This report and the contact sheet are the committed artifacts.

> Run note: a DNS outage hit the machine during the run's final full-image download
> phase; the script degraded gracefully (kept previous local files, 0 hard failures)
> and a targeted repair pass afterwards re-downloaded all 183 changed picks
> successfully (0 still failing) and recomputed the affected `color_family` values
> (8 changed). The script now also reverts `image_url` on a failed re-fetch so an
> outage can never wedge an item into a "registry says new, disk has old" state again.

## Churn stability

The picker's argmax disagreed with the currently-shipped image on **410 of 1264**
carried-over products (32%). The anti-churn stability rule suppressed **223** of
those disagreements (54%) as non-decisive preference flips between legitimate
alternates, and allowed **187** genuine replacements through.

Anti-churn margin threshold: **0.15** (picker FLOOR is 0.45). An image is only
swapped when the picker's score for the new pick exceeds the currently-shipped
image's own score by more than this margin -- and only when the shipped image itself
still clears FLOOR (a known-bad image is never protected by stability). Threshold
rationale (see `apply_stability_rule()` docstring): report 035's one measured
legitimate-alternate margin was 0.009, while its measured real-fix margins were
0.25-0.94 (maintainer cases) and >=0.28 (audit-component flip on the report-024
recovered set); 0.15 separates the two observed populations with room on both sides.

**Known false-positive class (2 products lost vs. the old registry):** an opaque
*white* sheet photographed on a white studio ground scores 0.37-0.43 -- the sheet
blends into the background (`fg_frac` 0.23-0.40, too high for the picker's
pale-sheet credit which requires <=0.02, too low to read as full-bleed coverage),
plus the weak `product_on_white` audit penalty. This is the pale-sheet weak spot
report 035 already documented as not stress-tested. The picker was deliberately NOT
retuned in this integration branch (its calibration is what was validated);
recommended follow-up on the research side: extend the pale-sheet credit to
intermediate `fg_frac` when the foreground is itself near-white, then restore these
two SKUs.
